# Research and provenance

## Source order

Use sources in this order:

1. Current user instructions and supplied materials.
2. Current QLander repository and local paths explicitly named by the user.
3. Supplied official website and its relevant official pages.
4. Broader public web sources only after approval.

Do not scan sibling directories or unrelated local projects automatically.

## Official website research

Inspect the smallest relevant set of official pages, normally the homepage plus applicable About, Services/Products, Contact, Locations, FAQ, Case Studies, Testimonials, Pricing, and policy pages. Follow same-organization official links when they materially fill the brief.

Capture direct source URLs for:

- name, description, contact information, locations, and social profiles
- audience, offers, services/products, pricing, process, and differentiators
- proof, testimonials, certifications, guarantees, and measurable claims
- navigation, tone, palette, logo, and available media

Summarize or rewrite source prose. Preserve exact factual values and short product/service names. Do not reproduce long passages.

## Local context

Inspect the current repository and user-named files/directories read-only before drafting. Prefer structured data, current specifications, approved brand files, product data, and recent project notes over stale generated output.

Do not place confidential notes or secrets into `content/site-brief.md`. Record a neutral public-safe summary or mark the item as requiring user input.

## Fact status

Assign every material fact one status:

- `user-provided`: explicitly supplied in the current request or approved materials
- `verified`: supported by a named local source or official URL
- `inferred`: a proposed interpretation that requires confirmation
- `needs-confirmation`: conflicting, absent, stale, or materially ambiguous

When sources conflict, show both values and their dates when available. Do not choose silently.

## Refresh

Record `lastReviewed` in the brief. Offer a refresh before major work when it is more than 30 days old, the official URL changed, the user mentions a rebrand/migration, or source-dependent facts may have changed. Routine edits reuse the existing brief.
